# 改善点メモ

StageLayout Designer の現状を見て、今後直すとよさそうな点を優先度別に整理します。
今回は大規模なリファクタリングは行わず、読みやすくするためのコメント追加に留めています。

## 優先度高

- `CanvasPanel` に描画、選択、移動、リサイズ、線描画、バミリ、テキストボックス、背景図面、undo/redo が集中している。
  すぐ分割すると危険ですが、今後不具合が増えやすい場所なので、描画処理、入力処理、履歴処理、背景図面処理を少しずつ切り出す候補です。
- モード切替は改善されてきたものの、文字、線、バミリ、会場パーツ、機材配置の状態が `CanvasPanel` と `MainFrame` にまたがっている。
  使っている人から見たモード表示と内部状態がずれると混乱するため、今後も変更時は重点確認が必要です。
- `.stage` 保存形式に後から列が増えているため、読み込み時の互換処理が重要になっている。
  新しい保存項目を足す時は、古いファイルを読めるか必ず確認する必要があります。
- プレビューは提出用として重要なので、作業画面と同じ描画を流用しすぎると見づらくなる。
  テキストボックス、バミリ、必要機材一覧などは、プレビュー専用の見せ方を守る必要があります。

## 優先度中

- テンプレート、背景図面、会場パーツの関係が少し複雑になっている。
  214教室などの標準テンプレート、PDF/画像背景テンプレート、ユーザー保存テンプレートを、利用者には同じ「会場の土台」として見せる整理が必要です。
- 必要機材一覧は全件表示に寄せていますが、件数が多い時は紙面内で読みづらくなる可能性がある。
  将来的には2ページ目や別紙出力も検討できます。
- `RoomObject` が四角、円、画像、楕円、トラック風などをtypeで持っている。
  形状が増え続ける場合は、描画だけを別クラスに分けると読みやすくなります。
- PDF/画像背景は元ファイルパスに依存している。
  別PCで開く時や配布時に背景が見つからない可能性があるため、テンプレート保存時のコピー方針を今後さらに整理すると安全です。
- ツールバーとメニューは整理されてきたが、機能が増えると再び入口が増えすぎる可能性がある。
  初心者向けには「よく使う操作」と「管理操作」を分ける方針を維持するとよいです。

## 優先度低

- UIの色、余白、ボタン名をさらに統一すると、アプリ全体の見た目が落ち着きます。
- コード製テンプレートの 214教室、大学野外ステージは便利ですが、将来的にはユーザー保存テンプレートへ寄せると引き継ぎやすくなります。
- 現在は手動コンパイル中心なので、将来的に簡単なビルドスクリプトを用意すると配布作業が楽になります。
- PDFBoxなど外部ライブラリのライセンス確認手順を、配布前チェックリストにしておくと安心です。
